# Design Skill

이 스킬은 study_blog 의 디자인 작업이 `src/styles/global.css` 의 "Quiet Tech v3" 토큰 시스템을 벗어나지 않도록 강제합니다. 새 컴포넌트를 만들거나 기존 스타일을 손볼 때, 색·폰트·간격·breakpoint 가 토큰으로만 표현되도록 가드 역할을 합니다.

`blog-draft` / `idea-listing` 스킬이 콘텐츠를 다룬다면, 이 스킬은 **시각 표현 일관성**을 다룹니다.

## 트리거

다음 셋 중 하나 이상이 감지되면 발동합니다.

### 1. 명시 호출

- `/design` 슬래시 커맨드
- `[design/...]` prefix 메시지 (예: `[design/Footer] 링크 hover 색을 더 진하게`)

### 2. 파일 경로 기반 자동 발동

다음 경로의 `.astro` / `.css` / `.scss` / `.tsx` / `.jsx` 파일을 편집·생성하라는 요청이 들어오면 자동 발동합니다.

- `src/components/**`
- `src/layouts/**`
- `src/styles/**`
- `src/pages/**`

### 3. 자연어 키워드 자동 발동

다음 단어가 사용자 메시지에 포함되면 자동 발동합니다.

- 시각 표현: "스타일", "컴포넌트", "레이아웃", "디자인"
- 색·폰트: "색상", "컬러", "폰트", "타이포", "글꼴"
- 공간: "간격", "여백", "패딩", "마진", "gap"
- 반응형: "반응형", "모바일", "breakpoint", "미디어쿼리"
- 모드: "다크모드", "라이트모드"

> 사용자가 명시적으로 "토큰 신경 쓰지 말고 일단 임시로 적어줘" 같이 가드 해제를 요구하면 이번 한 번만 가드를 풀고, 작업 결과 보고 시 "일회성 예외 적용"이라고 명시합니다.

## 선행 단계 — 토큰 로드 (필수)

작업 시작 *전*에 항상 다음을 수행합니다.

1. `src/styles/global.css` 의 `:root { ... }` 블록을 read
2. 사용 가능한 CSS 변수 목록을 컨텍스트에 적재
3. 새 세션이거나 직전 메시지 이후 토큰 파일이 변경됐을 가능성이 있으면 재로드

이 단계를 건너뛰고 작업하지 않습니다. 토큰을 모르면 가드도 작동할 수 없습니다.

### 현재 사용 가능한 주(primary) 토큰

> 이 표는 참고용 스냅샷입니다. **실제 작업 시에는 항상 `global.css` 를 직접 읽어 최신 상태를 확인**하세요.

| 영역 | 토큰 |
|---|---|
| 배경 | `--bg`, `--bg-elev`, `--bg-alt`, `--bg-code` |
| 테두리 | `--border`, `--border-strong`, `--border-hair` |
| 잉크(텍스트) | `--ink`, `--body`, `--muted`, `--subtle` |
| 강조 | `--accent`, `--accent-soft`, `--accent-ink` |
| 레이아웃 | `--content-w` (720px), `--h-scale` |
| 폰트 | `--font-sans`, `--font-mono` |

`--color-*` / `--bg-primary` / `--text-primary` 같은 legacy alias 도 정의돼 있지만, **새 코드에서는 주 토큰을 우선 사용**합니다. legacy alias 는 기존 컴포넌트 호환용으로만 둡니다.

## 하드 룰 (어김 금지)

| 영역 | 규칙 |
|---|---|
| 색상 | hex (`#xxx`) / `rgb()` / `rgba()` / `hsl()` 직접 사용 금지. 모두 `var(--accent)` 같은 기존 변수로. 매칭되는 변수가 없으면 작업 중단하고 사용자에게 신규 토큰 추가 여부 질문 |
| 폰트 | `font-family` 하드코딩 금지. `var(--font-sans)` / `var(--font-mono)` 만. 가중치는 400 / 500 / 600 / 700 / 800 만 사용 |
| 간격 | `padding` / `margin` / `gap` 은 4 · 8 · 12 · 16 · 24 · 32 · 48 · 64 px 의 8px grid 값만. 그 외 값은 사용자 확인 후 적용 |
| Breakpoint | `@media (max-width: 1024px / 720px / 420px)` 세 가지만. 새 breakpoint 도입 시 사용자 확인 |
| Max-width | 본문 영역은 `var(--content-w)` (720px). 와이드 레이아웃이 정말 필요할 때만 사용자에게 토큰 추가 제안 |
| 이모지 | 모든 이모지 금지. `§` `·` `→` `←` `─` `✓` `✗` `◇` `◆` `•` 같은 텍스트 기호로 대체 |
| 라이브러리 | 새 CSS 라이브러리·Tailwind plugin·shadcn 등 외부 컴포넌트 라이브러리 추가 금지. 기존 vanilla CSS + CSS 변수 + 기존 Tailwind utility 만 사용 |

이 룰을 어길 만한 코드를 사용자가 직접 요구하면, 자동 적용하지 말고 **이유를 설명하고 대안을 제시**합니다.

## 소프트 룰 (권장)

- Astro 컴포넌트는 `<style>` 블록 안에서 CSS 변수 참조 (`color: var(--ink)`)
- 호버·포커스 상태도 토큰 기반: `border-color: var(--accent)`, `background: var(--accent-soft)` 등
- 본문 글의 H2 `§` prefix 는 `.prose h2::before` 가 자동 적용. 새 컴포넌트에서 별도 prefix 를 직접 박지 말 것
- line-height 는 다음 셋 중 가까운 값 우선: `1.6` (UI 일반), `1.85` (본문), `1.7` (코드 블록)
- transition 은 `0.15s` 기본 (기존 `.prose a` 와 일치)

## 작업 흐름

1. **트리거 감지** — 위 "트리거" 절의 셋 중 하나 이상에 해당하는지 확인
2. **토큰 로드** — `src/styles/global.css` read, 변수 목록 추출
3. **요청 분석** — 사용자 의도 파악, 영향 받는 파일 식별
4. **가드 검증** — 작성 예정 코드에 하드 룰 위반이 있는지 사전 점검
   - 위반 0건 → 그대로 적용
   - 위반 발견 → 토큰으로 변환 시도. 매칭되는 토큰이 없으면 사용자에게 질문 (아래 "실패 모드" 참조)
5. **적용** — Edit / Write 로 변경
6. **결과 보고** — 아래 "결과 보고 형식" 따름

## 결과 보고 형식

작업 완료 후 항상 다음 정보를 출력합니다.

```
변경된 파일:
- src/components/Foo.astro

사용한 토큰:
- --bg, --ink, --accent

차단·변환한 하드코딩:
- #6366F1 → var(--accent)
- font-family: 'Inter' → var(--font-sans)

추가 제안 (있을 때만):
- 와이드 레이아웃용 --content-w-wide 토큰 도입 검토
```

차단 항목이나 추가 제안이 없으면 그 줄은 생략합니다.

## 실패 모드

다음 상황에서는 자의적으로 진행하지 말고 사용자에게 선택지를 제시합니다.

- **매칭되는 토큰이 없는 색·간격·폰트가 필요한 경우**:
  1. 기존 토큰 중 가장 가까운 것을 재사용
  2. `global.css` 에 신규 토큰 추가 (이름·용도 제안 포함)
  3. 일회성 예외로 하드코딩 (해당 라인에만 적용, 별도 메모)
- **breakpoint 1024 / 720 / 420 외의 분기가 필요한 경우**: 위와 동일하게 셋 중 하나 선택
- **이모지가 콘텐츠 자체로 필요하다고 사용자가 명시한 경우 (예: 이모지 라이브러리 데모)**: 그 페이지에 한해 예외 허용, 다른 곳으로 전파 금지

자의적으로 새 CSS 변수를 `:root` 에 추가하거나, breakpoint 를 임의로 도입하지 않습니다.

## 알려진 부채 (skill 외부 — 기록만, 자동 수정 안 함)

다음은 현 코드베이스의 흩어진 부분으로, 이 스킬이 발견 시 결과 보고에 "추가 제안"으로만 표기합니다. 임의로 손대지 않습니다.

- **다크모드 미지원** — `:root` 에 라이트 모드 토큰만 정의. `prefers-color-scheme` 분기 없음. 다크 토큰 세트 추가 시 별도 작업으로 분리
- **max-width 분산** — 컴포넌트마다 `720` / `1200` / `max-w-5xl` 가 혼재. `--content-w-wide` 같은 토큰을 추가해 통일하면 좋음
- **미디어쿼리 분산** — 컴포넌트마다 `@media` 가 흩어져 있음. 공용 mixin·CSS layer 도입 검토
- **legacy alias 다수** — `--color-*` / `--bg-primary` 등이 호환용으로 남아 있음. 사용처를 점진적으로 주 토큰으로 마이그레이션

## 작성 시 톤 (사용자와의 대화)

- 경어체 일관 (`~합니다`, `~했어요`, `~로 보입니다`)
- AI 클리셰 금지: "훌륭한 질문입니다", "정확하십니다", 빈 사과, 자기 내레이션 남발, "AI로서·언어 모델로서" 자기언급
- 이모지 사용 금지 (스킬 자체 출력에도)
- 단정보다 트레이드오프 노출. "이 토큰을 쓰는 게 일관성 측면에서 더 낫습니다" 식으로

## 커밋 컨벤션

`blog-draft` / `idea-listing` 과 달리 이 스킬은 **자동 push 하지 않습니다**. 사용자가 직접 push 합니다.

```
feat(design): {짧은 설명}     # 새 컴포넌트·새 스타일 도입
style(design): {짧은 설명}    # 토큰 정리·기존 스타일 미세 조정
fix(design): {짧은 설명}      # 디자인 버그·깨진 레이아웃 수정
refactor(design): {짧은 설명} # 하드코딩 → 토큰 마이그레이션
```

규칙:
- 한 작업 = 한 커밋
- 한 커밋 안에 여러 컴포넌트·여러 영역 변경을 섞지 말 것
- 영향 받는 컴포넌트 명을 scope 에 더 좁힐 수 있으면 좁힘 (예: `style(design/Footer): 링크 호버 색을 accent-ink 로`)

## 작업하지 않을 것

- `global.css` 의 `:root` 토큰 임의 추가·수정 (사용자 승인 필요)
- 다른 프로젝트(글로벌 `~/.claude/skills/`)로 이 스킬 복제
- Tailwind 도입·제거·plugin 추가
- 기존 컴포넌트의 하드코딩 일괄 리팩토링 (요청 들어온 파일만 처리)
- 이미지·SVG·아이콘 라이브러리 임의 도입
- 디자인 결정의 정답 단언 — "이게 더 좋은 디자인입니다" 가 아니라 "토큰 일관성 측면에서는 이쪽" 으로
